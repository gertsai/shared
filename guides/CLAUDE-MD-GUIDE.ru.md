# CLAUDE-MD-GUIDE — как писать CLAUDE.md учитывая слабости LLM

Гайд по проектированию файла `CLAUDE.md` (он же system prompt для Claude Code).
Основан на известных ограничениях моделей трансформер-семьи и эмпирических результатах
аудита CLAUDE.md Forgeplan (2026-04-17).

---

## TL;DR (цели)

- **Объём**: ≤400 строк, ≤7K токенов
- **Структура**: primacy → справочник → recency
- **Маркеры**: ≤7 "красных линий", 0 избыточных CAPS
- **Дубли**: 0 — один workflow, одна формулировка
- **Case studies**: в `docs/methodology/LESSONS.md`, не в CLAUDE.md

---

## Часть 1 — Слабости модели, которые нужно учитывать

### 1.1 U-кривая внимания ("Lost in the Middle")

**Что это**: В длинных контекстах LLM значительно лучше извлекают информацию из
начала и конца, чем из середины. Кривая attention напоминает букву U.

**Эмпирическое исследование**: Liu et al. 2023 (Lost in the Middle) показали
что при контексте 30+ документов precision в извлечении середины падает на 20-30%
относительно начала/конца.

**Следствие для CLAUDE.md**:
- Самые критичные правила → **первые 80 строк** (primacy zone)
- Контекст проекта / справочники → середина
- Non-Goals / финальные ограничения → **последние 40 строк** (recency zone)

**Как это проявляется на практике**:
> В Forgeplan до аудита Git-политика сидела на строках 388-525 (точно середина).
> Результат: пришлось завести 4 отдельных feedback-memory записи (`never_commit_to_dev`,
> `squash_merge_loss`, `never_push_without_review`, `fmt_before_commit`) — модель
> пропускала эти правила при выполнении задач.

**Починка**: перенести все критичные необратимые правила в "🔴 КРАСНЫЕ ЛИНИИ" в начале.

---

### 1.2 Cry-wolf effect (инструкционная усталость)

**Что это**: Когда CAPS-маркеры, "CRITICAL", "ОБЯЗАТЕЛЬНО", "НИКОГДА" используются
на каждые 20-30 строк, их сигнальная ценность обнуляется. Модель учится "это просто
стиль документа, а не уровень приоритета".

**Порог**: после 5-7 "критичных" маркеров на файл каждое следующее правило воспринимается
как обычное guideline.

**Как это проявляется**:
> CLAUDE.md Forgeplan имел 17× "ОБЯЗАТЕЛЬНО" и 10× "CRITICAL". Feedback-memory
> содержала записи с повторным усилением: `feedback_fmt_before_commit.md` маркирован
> CRITICAL — т.е. обычного "ОБЯЗАТЕЛЬНО smoke test" не хватало, пришлось усиливать.

**Правило**:
- **≤5 красных линий** в файле — только для необратимых последствий
- "Обязательно", "критично" без нужды → обычный тон
- Если захотелось пометить правило CAPS — спросить: "а что из уже CAPS я готов понизить?"

---

### 1.3 Context dilution

**Что это**: Длинный system prompt снижает instruction-following. На 13K токенов
system prompt модель тратит часть attention budget просто на удержание контекста —
меньше остаётся на задачу пользователя.

**Практическое правило**: Claude Code system prompt + CLAUDE.md + tools description
+ user message уже могут доходить до 30-50K токенов. Если CLAUDE.md = 13K, это
≥25% общего бюджета. Сократить до 5-7K значит освободить до 15% attention для задачи.

---

### 1.4 Redundancy cost

**Что это**: Повторение одного правила разными словами НЕ усиливает его — оно
**ослабляет** его, потому что:
1. Съедает attention budget (см. 1.3)
2. Создаёт ambiguity: "три формулировки — какой следовать?"
3. Провоцирует inconsistency: одна из версий содержит опечатку / старый вариант

**Типичные дубли** (по аудиту Forgeplan):
- "Полный цикл" (70-83) + "PR pipeline" (496-511) + "smoke test" (120-151, 576-590)
- "Структура проекта" (592) + "Storage:" (737) пересекались на 60%
- Правило `forgeplan init -y` встречалось 4 раза в разных местах

**Правило**: один концепт → одна формулировка → один раздел. Всё остальное — ссылки.

---

### 1.5 Specificity trap / overfitting

**Что это**: Детальный case study в system prompt (с именами веток, PR-ID, датами)
заставляет модель пытаться применить этот паттерн к несвязанным ситуациям.

**Пример**:
> В CLAUDE.md Forgeplan сидел детальный разбор "Sprint 13.1.5 dependent branch verification"
> с датами и PR-ID. Риск: при любой работе с branches модель начинала упоминать
> Sprint 13.1.5 паттерн, даже если контекст не про dependent branches.

**Правило**: в CLAUDE.md — обобщённое правило в одну строку. Детальные кейсы —
в `docs/methodology/LESSONS.md`.

---

### 1.5.1 Context priming — явные правила, а не просто checklist

**Что это**: "Прогрев контекста" — это загрузка минимально-необходимых данных в начале
сессии, чтобы модель могла принимать релевантные решения без постоянного "а где это?".

**Типичная ошибка в CLAUDE.md**: описать *что делать* ("запусти три команды"), но не
описать *как принимать решения* — в каком порядке, что делать при fallback, когда
re-warm, как понять что контекста достаточно.

**Разница между checklist и protocol:**
- Checklist: "запусти X, Y, Z"
- Protocol: "запусти X, Y, Z параллельно; если Y недоступен — продолжай; re-warm когда
  переключаешься на новую область; критерий достаточности — можешь назвать
  {ключевые сущности}"

**4 обязательных компонента любого context-priming раздела:**

1. **Три источника** (минимум) — long-term memory / current state / ongoing work.
   Меньше трёх — blind spots; больше трёх — overload.
2. **Что НЕ читать на старте** — anti-overload правило. Обычно это длинные файлы
   (CHANGELOG, ROADMAP, full TODO) — читать их только когда релевантны вопросу.
3. **Re-warm триггеры** — когда посреди сессии подгружать ещё контекст
   (переключение темы, переход на новую фичу, появление незнакомой области).
4. **Критерий достаточности** — как модель понимает "хватит читать, можно работать".
   Обычно формулируется как "можешь назвать X, Y, Z".

**Почему это важно**:
- Без fallback правила: модель стопорится когда один из MCP недоступен
- Без anti-overload: модель читает весь CHANGELOG "на всякий случай" → жжёт 10K токенов
- Без re-warm триггеров: модель не подгружает новый контекст при смене темы, выдаёт стейл-ответы
- Без критерия достаточности: модель либо недочитала, либо перечитала

**Особый случай — auto-loaded context**:

Если система подгружает какой-то файл автоматически (у Claude Code это `MEMORY.md`),
это нужно **явно указать** в CLAUDE.md. Иначе модель будет избыточно вызывать
`memory_recall` "на всякий случай", не зная что индекс уже в контексте. Это реальная
проблема, замеченная в Forgeplan до правки.

Пример правильной формулировки:
> "`MEMORY.md` is auto-loaded every turn — no command needed. Call `memory_recall`
> only for records beyond the index."

### 1.6 Conflicting instructions

**Что это**: Когда два правила в CLAUDE.md противоречат друг другу (или противоречат
инструкциям Claude Code по умолчанию), модель делает локальный выбор и это поведение
становится недетерминированным.

**Типичные конфликты**:
- "Не создавай новые файлы" (default) vs "создай ARCHITECTURE.md в корне"
- "Не используй emoji" (default) vs "используй 🔴🟡🟢 для зон"
- "Коротко отвечай" (default) vs "детально документируй каждый шаг"

**Правило**: если нужен override default — явно назвать: "в этом проекте emoji
разрешены в заголовках для visual hierarchy, но не в теле текста".

## Часть 2 — Структура хорошего CLAUDE.md

```
┌─ PRIMACY ZONE (строки 1-80) ─────────────┐
│  1. Title + язык проекта                  │
│  2. 🔴 КРАСНЫЕ ЛИНИИ (≤7 пунктов)        │
│  3. Что за проект (3-5 предложений)       │
│  4. Текущий статус + ссылки на CHANGELOG  │
│  5. Session start (3 команды)             │
│  6. Полный цикл (single source of truth)  │
└───────────────────────────────────────────┘

┌─ REFERENCE ZONE (строки 80-~280) ────────┐
│  7. Routing table                         │
│  8. Domain-specific rules (таблицы!)      │
│  9. Git краткие правила                   │
│ 10. Permissions / Safety                  │
│ 11. Coding rules                          │
│ 12. AI-агентам (non-interactive)          │
│ 13. Artifact / Model types                │
│ 14. Ключевые формулы                      │
│ 15. Storage layout                        │
│ 16. Архитектура (короткая карта)          │
└───────────────────────────────────────────┘

┌─ RECENCY ZONE (строки ~280-400) ─────────┐
│ 17. Smoke test                            │
│ 18. Unified Workflow / Integration        │
│ 19. Non-Goals (что НЕ делаем)             │
└───────────────────────────────────────────┘
```

### Почему Non-Goals в конце

Non-Goals — это "стоп-сигнал" против scope creep. Модель запомнит его из recency zone
и **не предложит** функциональность, которая не в scope проекта. Это ценный filter.

## Часть 3 — Чеклист перед коммитом CLAUDE.md

```markdown
## Структура
- [ ] ≤400 строк, ≤7K токенов (wc -l, wc -c)
- [ ] Красные линии в начале (первые 80 строк)
- [ ] Non-Goals в конце
- [ ] Workflow описан один раз, не дублируется

## Context priming (session start)
- [ ] Раздел "Session Start" существует, находится в primacy zone (до workflow)
- [ ] Описаны минимум 3 источника (memory / workspace / tasks)
- [ ] Есть правило "что НЕ читать на старте" (anti-overload)
- [ ] Описаны re-warm триггеры (что делать при смене области)
- [ ] Сформулирован критерий достаточности ("когда контекста хватит")
- [ ] Описан fallback — что делать если источник недоступен
- [ ] Auto-loaded файлы (MEMORY.md и т.п.) явно помечены — чтобы модель не дёргала MCP зря

## Маркеры
- [ ] ≤7 "красных линий"
- [ ] ≤5 "ОБЯЗАТЕЛЬНО/CRITICAL/НИКОГДА" во всём файле
- [ ] Если CAPS — оправдано (необратимое, blast radius, shared state)

## Дубли (grep-проверка)
- [ ] `grep -c "smoke test"` ≤1
- [ ] `grep -c "ОБЯЗАТЕЛЬНО"` ≤5
- [ ] `awk '/^## /{print}'` — нет повторяющихся секций
- [ ] Git правила не размазаны по 3 местам

## Case studies
- [ ] Нет детальных разборов Sprint N.M в CLAUDE.md
- [ ] Все incident lessons → `docs/methodology/LESSONS.md`
- [ ] Все полные workflows → `docs/operations/*.md`

## Проверка конфликтов
- [ ] Ни одно правило не противоречит default Claude Code
- [ ] Если override — явно прокомментирован
- [ ] Нет скрытых "вcе это для эстетики" правил (emoji, форматирование)

## Читаемость
- [ ] Таблицы вместо прозы где можно
- [ ] Команды в code blocks (не в тексте)
- [ ] Ссылки на полные доки там, где короткого правила достаточно
```

## Часть 4 — Анти-паттерны

### ❌ "Ещё один смоук-тест"

```markdown
### ОБЯЗАТЕЛЬНО smoke test после каждого спринта:
...
### ОБЯЗАТЕЛЬНО smoke test после релиза:
...
### ОБЯЗАТЕЛЬНО smoke test перед PR:
```

Три копии одного теста. Результат: модель не знает какой запустить.
**Фикс**: один smoke test раздел, варианты — через ссылки ("вариант для релиза: см. LESSONS.md L-NNN").

### ❌ "Всё критично"

```markdown
### ОБЯЗАТЕЛЬНО session start
### ОБЯЗАТЕЛЬНО fmt before commit
### ОБЯЗАТЕЛЬНО test every pub fn
### КРИТИЧНО never push without review
### ВАЖНО check health
### ОБЯЗАТЕЛЬНО audit before PR
### НИКОГДА not commit to dev
```

7 разных уровней "обязательности" неразличимы. Модель выбирает что ей удобнее.
**Фикс**: 3-5 красных линий, остальное — "правила" без маркера.

### ❌ "History inside system prompt"

```markdown
#### КРИТИЧНО: Dependent sprint base (урок Sprint 13.1.5, 2026-04-07)
... 25 строк детального разбора с PR-ID и коммитами ...
```

Модель overfits на Sprint 13.1.5 паттерн.
**Фикс**: в CLAUDE.md — одна строка "проверяй base перед dependent sprint: `git log ... | grep ...`. Детали: LESSONS.md". Полный разбор — в LESSONS.md.

### ❌ "Mixed languages without purpose"

```markdown
## Core workflow
1. Run `cargo test` — ЕСЛИ fail — НЕ коммитить
2. After test passes, execute `cargo fmt`
3. Затем сделать commit с Refs на артефакт
```

Непредсказуемое переключение между EN/RU. Модель может начать отвечать в любом.
**Фикс**: выбрать один язык инструкций. Код/команды — на оригинальном (EN). Body коммитов — на языке документации. Явно указать в начале файла.

### ❌ "Тест-дескрипшны без правил"

```markdown
### Rules
- **Refs обязательны** — каждый коммит ссылается на артефакт
- **Один коммит = одна логическая единица** — не мешать feat + docs
- **Description на английском**, body на русском
```

Три правила кучно, без иерархии. При потере attention модель забудет самое важное.
**Фикс**: сначала MUST (красная линия), потом SHOULD (guideline), потом NICE-TO-HAVE.

## Часть 5 — Миграция существующего CLAUDE.md

Если у вас уже есть раздувшийся CLAUDE.md:

```bash
# 1. Измерить
wc -l CLAUDE.md && wc -c CLAUDE.md
grep -c "ОБЯЗАТЕЛЬНО\|CRITICAL\|НИКОГДА" CLAUDE.md

# 2. Backup
cp CLAUDE.md .local/CLAUDE.md.bak.$(date +%Y%m%d-%H%M%S)

# 3. Найти дубли
awk '/^## /{print NR": "$0}' CLAUDE.md   # H2 заголовки
awk '/^### /{print NR": "$0}' CLAUDE.md  # H3 заголовки
# ищи одинаковые названия

# 4. Выделить в docs/
#    - Git workflow → docs/operations/GIT-WORKFLOW.md
#    - Incident stories → docs/methodology/LESSONS.md
#    - Full methodology → docs/methodology/*

# 5. Переписать по шаблону
cp templates/claude-md/CLAUDE_TEMPLATE.md CLAUDE.md
# заполнить placeholder'ы

# 6. Проверить чеклистом (Часть 3)

# 7. Измерить результат
wc -l CLAUDE.md   # ожидаемо ≤400
```

## Часть 6 — Почему это работает (короткая теория)

### Attention — это ограниченный ресурс

Трансформер распределяет attention по всему контексту. Чем длиннее промпт, тем
"тоньше" attention на каждую точку. Короткий, хорошо структурированный промпт
= высокая концентрация attention на важном.

### Primacy/Recency = архитектурные свойства, не баг

U-кривая — не баг конкретной модели, это следствие positional encoding и
training-distribution transformer'ов. Её нужно не "лечить" попыткой равномерно
распределить вес, а **эксплуатировать**: класть важное на края.

### Cry-wolf — это rational behaviour модели

Когда 20% строк помечены "CRITICAL", модель корректно выводит статистически:
"CRITICAL здесь = distribution feature, не priority signal". Это правильный
вывод на основании данных. Чтобы CRITICAL работало — держи его редким.

### Single Source of Truth снижает ambiguity

Если одно правило описано в 3 местах разными словами, модель в каждом эпизоде
выбирает случайную версию. Результат — недетерминированное поведение.
Один раз — детерминированно.

## Часть 7 — Метрики успеха

После переписывания CLAUDE.md проверить:

| Метрика | До | После | Цель |
|---|---|---|---|
| Строки | ≥600 | ≤400 | ≤400 |
| Символы | ≥30K | ≤15K | ≤15K |
| Токены (~chars/3.5) | ≥8K | ≤5K | ≤5K |
| "ОБЯЗАТЕЛЬНО" | ≥10 | ≤3 | ≤5 |
| "CRITICAL" | ≥5 | ≤2 | ≤2 |
| Дубли H3 | ≥2 | 0 | 0 |
| feedback-memory: новые правила в git/workflow | растёт | стабильно | стабильно |

Последняя метрика — самая важная. Если после переписывания CLAUDE.md **прекращают
появляться** feedback-memory записи про те же правила — значит модель их теперь
видит и следует.
